# LLVMPass -- def-use граф с райнтайм-значениями

Это LLVM-плагин, который для произвольной `С` программы:
- Строит def-use граф цепочек в формате graphviz;
- Логирует свои инструкции;
- После запуска накладывает реальные значения, полученные в рантайме, на узлы графа.

В зависимости от того, какие данные подаются на вход программе, итоговая картина получается разной -- отражается не только сама структура, но и проход исполнения.

## Структура

## Входная точка
`src/MyPass.cpp` -- это точка входа LLVM-пасса. В нем: 

- Определён класс `MyPass`, наследник `PassInfoMixin<MyPass>`, с методом `run(Module &M, ModuleAnalysisManager &)` -- именно его вызывает `opt` для каждого модуля.
- Реализована функция `llvmGetPassPluginInfo()`, которую `opt` находит при загрузке `.so` через `dlsym` и через которую плагин регистрирует имя пасса `my-pass` в `PassBuilder`.
Сама логика обработки IR вынесена в три модуля: `DotWriter` (запись графа), `Instrumenter` (вставка инструментации), `ValueIds` (выдача id).

## Что делает пасс верхнеуровнево?

Пасс является модульным: он получает на вход весь `Module Module` и проходит по всем определенным в нем функциям таким образом:

```
Module -> Function -> BasicBlock -> Instruction
```

и собирает 2 разных типа информации.

### Тип 1: построение графа

Пасс ищет 3 вида узлов:
- **`Argument`** -- параметры функции. Это "входы" данных в функцию: значения, которые никто не вычислял внутри, они пришли извне;
- **`Instruction`** -- все инструкции IR (`add`, `load`, `store`, `call`, `mul`, ...). Каждая инструкция в SSA-форме определяет ровно одно значение;
- **`ConstantInt`** -- константы. Они учитываются в лейблах (если попадаются как операнды), но **узлами в графе не становятся**, чтобы не завалить значениями картинку. Только реально вычисляемые значения получают узлы.

Для каждой подходящей инструкции пасс смотрит её операнды (`Instruction.operands()`). Для каждого операнда, если он `Instruction` или `Argument`, рисуется ребро `operand -> Instruction`.

### Тип 2: инструментация

Параллельно пасс модифицирует IR. Он:
 
- Объявляет в модуле две внешние функции -- `__my_log_i32(i32, i32)` и `__my_log_i64(i32, i64)`. Эти функции потом подцепятся из `Runtime.c` на стадии линковки.
- Идёт по всем инструкциям и отбирает те, что подходят для инструментации.
Подходящие -- это инструкции, которые:
- - имеют ненулевой результат типа `i32` или `i64` (для других у нас нет log-функции);
- - не являются терминаторами (`ret`, `br`, `switch` -- после терминатора в блоке не вставить);
- - не являются `PHINode` (PHI должны идти в начале блока, и вставка вызова между ними сломает IR);
- - не принадлежат самому runtime.
После каждой такой инструкции через `IRBuilder` вставляется `call void @__my_log_iXX(i32 <id>, iXX %result)`.

```llvm
  %9 = load i32, ptr %3, align 4
  call void @__my_log_i32(i32 12, i32 %9)
```

## Из чего строится граф
 
Граф пишется в `dots/graph.dot` в формате graphviz. На нём:
 
**Узлы** -- это значения IR. Каждый узел подписан так:
 
```
id=47
%add = add
values: 8
```
 
- **`id=N`** -- внутренний уникальный идентификатор, который пасс выдал этому значению. Используется только для связи с логом, никакого смысла в IR не имеет. Каждый `Value` получает свой id один раз и хранится в `std::unordered_map<const Value*, int>`.

- **`%имя = опкод`** -- короткая форма инструкции. `%имя` -- это имя результата в LLVM IR, такое же, какое можно увидеть в `.ll` файле. `опкод` -- название операции (`add`, `load`, `mul`, `call`, ...).
- **`%?`** -- заглушка для безымянных значений. LLVM не у всех инструкций даёт человеческое имя; если результат используется только локально, ему присваивается числовое имя (`%10`, `%11`...).
- **`arg %a`** -- это узел-параметр функции. Выделен жёлтым (`fillcolor=lightyellow`), чтобы визуально отличать «вход» от вычисляемых значений. `%a` -- имя параметра. Если параметр безымянный -- будет `arg %?`.
- **`const N`** -- встречается только внутри текстовых лейблов; отдельных узлов для констант я не делала (как уже упоминала ранее).
**Рёбра** направлены от "родителя" к "ребенку". Если есть `%add = add %a, %b`, то рисуются `arg %a -> %add` и `arg %b -> %add`.

**Кластеры**. Каждая функция оборачивается в `subgraph cluster_<name>` с пунктирной рамкой и подписью. Это группирует узлы по функциям и делает картинку читаемее. 
> Между функциями рёбер нет -- пасс не делает межфункционального анализа (например, `call` в `main` -- это просто узел, не связан с инструкциями внутри `Compute`).

### Как source-код превращается в граф
 
Возьмём наш `tests/test.c`:
 
```c
int Compute(int n, int b) {
    int sum = 0;
    for (int i = 0; i < n; i++) {
        sum = sum + b;
    }

    return sum;
}
 
int main(void) {
    int r = Compute(4, 7);
    printf("%d\n", r);
    return 0;
}
```

После `clang -emit-llvm -O0` функция `Compute` превращается в IR (приведено как есть, только с комментариями для читаемости):
 
```llvm
define i32 @Compute(i32 %0, i32 %1) {
  %3 = alloca i32                      ; место для n
  %4 = alloca i32                      ; место для b
  %5 = alloca i32                      ; место для sum
  %6 = alloca i32                      ; место для i
  store i32 %0, ptr %3                 ; сохраняем входы в стек
  store i32 %1, ptr %4
  store i32 0, ptr %5                  ; sum = 0
  store i32 0, ptr %6                  ; i = 0
  br label %7
 
7:                                     ; шапка цикла: проверка условия
  %8  = load i32, ptr %6               ; читаем i
  %9  = load i32, ptr %3               ; читаем n
  %10 = icmp slt i32 %8, %9            ; i < n ?
  br i1 %10, label %11, label %18
 
11:                                    ; тело цикла
  %12 = load i32, ptr %5               ; читаем sum
  %13 = load i32, ptr %4               ; читаем b
  %14 = add nsw i32 %12, %13           ; sum + b
  store i32 %14, ptr %5                ; записываем обратно в sum
  br label %15
 
15:                                    ; инкремент счётчика
  %16 = load i32, ptr %6               ; читаем i
  %17 = add nsw i32 %16, 1             ; i + 1
  store i32 %17, ptr %6                ; записываем обратно в i
  br label %7
 
18:                                    ; выход
  %19 = load i32, ptr %5               ; финальное значение sum
  ret i32 %19
}
```

В графе для `Compute` появятся:
 
- 2 жёлтых узла **`arg %?`** -- это `%0` и `%1`, аргументы функции (n и b);
- 4 узла `%? = alloca` -- слоты в стеке для 4 локальных переменных;
- много узлов `%? = load`, `%? = add`, `%? = icmp` -- собственно вычисления;
- узлы `store`, `br` -- терминаторы и `void`-инструкции, рёбра в них приходят, но в логе их нет (нечего логировать).

При вызове `Compute(4, 7)` (как в `main`) цикл выполнится 4 раза. Каждая итерация прогонит выделенные инструкции заново -- об этом следующий раздел.
 
## Как фиксируются и отображаются рантайм-значения
 
### Что происходит при запуске
 
`Runtime.c` определяет четыре функции:
 
- `__my_log_init` с атрибутом `constructor` -- вызывается автоматически **до** `main`. Открывает `runtime_log.txt` на запись.
- `__my_log_finish` с атрибутом `destructor` -- вызывается **после** `main` или при `exit()`. Закрывает файл.
- `__my_log_i32(int id, int32_t value)` и `__my_log_i64(int id, int64_t value)` -- пишут в лог строку `<id> <value>`.
Каждый раз, когда выполнение в инструментированной программе доходит до вставленной инструкции `call void @__my_log_i32(i32 N, i32 %X)`, в файл записывается строка `N <значение_X_в_момент_выполнения>`.
 
Финальный `runtime_log.txt` -- это пары `(id, value)` ровно в том порядке, в каком выполнялись инструкции.

### Как накладываются значения на граф
 
`scripts/Annotate.py` делает три вещи:
 
1. Читает `runtime_log.txt`, группирует значения по id: для каждого id строит список значений в порядке появления (`defaultdict(list)`).
2. Читает `graph.dot` построчно. Для строк вида `n<id> [label="id=<id>\n..."]` вытаскивает id и дописывает к лейблу строку `values: v1, v2, v3 ... (+N more)`.
3. Пишет результат в `graph_annotated.dot`.
В лейбле отображаются первые `SHOW_FIRST_N = 3` значения. Если их больше -- добавляется хвост `... (+X more)`. Если id ни разу не появился в логе -- узел получает пометку `values: (not executed)` (например, для инструкций в недостижимых ветках).
 
### Что если одна и та же переменная присваивается несколько раз
 
Ответ зависит от того, что именно в исходном C-коде называется «одной и той же переменной» и **как** на это смотрит LLVM IR.

#### Переменная -- это `alloca` на стеке
 
В нашем `Compute` переменная `sum` присваивается несколько раз:
 
```c
int sum = 0;            // первое присваивание
for (...) {
    sum = sum + b;      // и потом ещё n раз внутри цикла
}
```
 
В IR это выглядит так:
 
```llvm
%5 = alloca i32                ; ОДНА ячейка на стеке для sum
store i32 0, ptr %5            ; sum = 0
...
%12 = load i32, ptr %5         ; читаем текущий sum
%14 = add nsw i32 %12, %13     ; sum + b
store i32 %14, ptr %5          ; записываем новый sum
...
%19 = load i32, ptr %5         ; финальное чтение перед return
```
 
Что мы видим:
- **одна** `alloca %5` -- одно место в памяти, представляющее «переменную sum»
- **много разных** `store` -- каждый со своим уникальным id
- **много разных** `load` -- каждый со своим id
В графе будет один узел-`alloca` для `%5`, в который сходятся стрелки от нескольких `store`, а от него идут стрелки в несколько `load`. И когда программа запустится, runtime запишет в лог отдельные значения для каждого `load` -- потому что в IR это разные инструкции, хоть в C-коде это и «одна переменная sum».

Annotate-скрипт сгруппирует значения по id и покажет в лейбле:
 
```
id=...
%? = add
values: 7, 14, 21 ...
```

## Как запустить
 
Требования: macOS или Linux, Homebrew LLVM (тестировалось на LLVM 22), graphviz, Python 3.
 
```bash
brew install llvm graphviz cmake ninja
export PATH="$(brew --prefix llvm)/bin:$PATH"
 
# собрать плагин
mkdir -p build && cd build
cmake -G Ninja \
  -DCMAKE_C_COMPILER=$(brew --prefix llvm)/bin/clang \
  -DCMAKE_CXX_COMPILER=$(brew --prefix llvm)/bin/clang++ \
  -DLLVM_DIR=$(brew --prefix llvm)/lib/cmake/llvm \
  ..
ninja
cd ..
 
# полный конвейер
./scripts/run.sh
```
 
После прогона `./scripts/run.sh`:
- `dots/graph.dot` -- структура графа
- `dots/mapping.txt` -- таблица id <-> инструкция
- `dots/runtime_log.txt` -- лог значений
- `dots/graph_annotated.dot` -- граф с наложенными значениями
- `images/graph.png` -- структура
- `images/graph_annotated.png` -- финальная картинка
